# npv-build 2.0 ŌĆö Design Spec: GUI-First, Current-Patch, Cross-Platform Release

*Status: draft for review ┬Ę Date: 2026-07-17 ┬Ę Source: [IMPROVEMENT_REPORT.md](../../../IMPROVEMENT_REPORT.md)*

---

## 1. Background and Goals

npv-build converts a Cyberpunk 2077 save into an installable mod that spawns the player's V as an AMM NPC. Version 1.x is a Linux-developed, CLI-first, single-shot pipeline targeting game patch 2.13. The improvement report identified three strategic problems (stale patch target, possibly-obsolete donor/injector hacks, untested core) plus hygiene and security gaps.

### Goals

- **G1 ŌĆö GUI is the primary interface.** Every workflow is completable in the GUI without touching a terminal. The CLI remains as a secondary automation/testing surface.
- **G2 ŌĆö Current-patch support.** Saves from game patch 2.2 through 2.31 build correctly; 2.13 saves keep working.
- **G3 ŌĆö All report improvements implemented** (P0, P1, P2), tested.
- **G4 ŌĆö Multi-platform release.** Distributable bundled apps for Windows and Linux that require no preinstalled Python.

### Non-Goals

- macOS support (game/tooling not natively available).
- REDmod packaging format (plain .archive + AMM lua only).
- In-game appearance editing (Appearance Creator Mod interop may come later).
- Code-signing of Windows binaries in the first release (SmartScreen warning accepted; documented).

## 2. Decisions

| # | Decision | Choice | Rationale |
|---|---|---|---|
| D1 | Distribution | Bundled apps: PyInstaller one-dir ŌåÆ Windows `.zip`, Linux AppImage. External tools (WolvenKit.CLI, Blender, .NET if still needed) auto-downloaded on first run with checksum verification | GUI-first users must not need Python; keeps repo free of third-party binaries |
| D2 | CLI fate | Keep as thin secondary frontend over the shared core layer | Near-zero extra cost; enables automation and E2E tests |
| D3 | ArchiveXL | Time-boxed spike with decision gate; fallback to current donor+injector design if the spike fails | Highest-value simplification but unproven for this exact use case |
| D4 | Patch 2.13 | Keep supported alongside 2.3x via versioned parsing/tables | Tables exist; small cost |
| D5 | Tooling | `uv` (lockfile) + `ruff` (lint/format) + type checking in CI; hatchling backend | 2026 consensus stack per report |
| D6 | Spec location | New docs live under `docs/`; the six legacy root spec files are consolidated/archived under `docs/legacy/` | Report ┬¦1.1 repo hygiene |

- **CORE-1**: Extract a `PipelineService` API from the current orchestrator/`gui_backend`: `build(BuildRequest, on_event, cancel_token) -> BuildResult`. Both frontends consume only this API. `gui.py` keeps importing zero pipeline modules directly.
- **CORE-2**: The pipeline is expressed as named stages with declared inputs/outputs: `parse_save`, `resolve_mapping`, `bake_head`, `author_entity`, `assemble`, `inject` (branch B only), `pack`, `emit_amm_lua`.
- **CORE-3 ŌĆö Checkpointing**: each stage persists its output plus a manifest entry (stage name, input hashes, tool versions) in the build directory. `resume=True` reuses checkpoints whose input hashes match; otherwise the stage re-runs. GUI exposes this as "Retry from failed stage"; CLI as `--resume`.
- **CORE-4 ŌĆö Cancellation**: a `cancel_token` checked between stages and passed into adapters, which terminate their child process on cancel.

### 3.2 Adapters (subprocess policy)

- **ADP-1**: All external process invocations go through adapter classes in `core/adapters/` (extending the existing `wk_cli.py` pattern). Direct `subprocess.run` calls in `part_resolver.py`, `wolvenkit.py`, `blender_module.py`, `installer.py` are removed.
- **ADP-2**: Every invocation has a timeout (defaults per tool class: WolvenKit uncook/pack 600 s, Blender bake 900 s, small conversions 120 s; all configurable) and raises a structured error (`ToolError` with tool name, argv, exit code, tail of output) on failure or timeout.
- **ADP-3**: Adapters emit log records for start/finish/duration so the GUI progress view is fed uniformly.

### 3.3 Error model

- **ERR-1**: Exception hierarchy rooted at `NpvError(user_message, remediation, details)`. Subtypes: `SaveFormatError`, `UnsupportedPatchError`, `MappingError`, `ToolError`, `BakeVerificationError`, `InstallError`, `SecurityError`.
- **ERR-2 ŌĆö Hard-fail enforced**: audit all 45 broad `except` sites (report ┬¦1.2). Each becomes: narrowed to expected exception types, re-raised as a typed `NpvError`, or deleted. No handler may silently continue with degraded output. `part_resolver.py`'s 20 sites are the priority.
- **ERR-3**: GUI renders `NpvError.user_message` + remediation in an error dialog with "Show log" / "Export debug bundle" actions. CLI prints the same fields and exits non-zero.

### 3.4 Logging

- **LOG-1**: Replace all 109 `print()` calls with the `logging` module; per-module loggers.
- **LOG-2**: CLI `-v/-vv` maps to INFO/DEBUG; `--log-file` supported. Every GUI build always writes a timestamped log file in the build directory.
- **LOG-3**: GUI attaches a handler that streams records into the progress/log pane; no stdout scraping.

### 3.5 Platform layer

- **PLT-1 ŌĆö Save discovery** (replaces hardcoded path at `gui.py:549`):
  - Windows: `%USERPROFILE%\Saved Games\CD Projekt Red\Cyberpunk 2077\`
  - Linux Steam/Proton: `<steam-library>/steamapps/compatdata/1091500/pfx/drive_c/users/steamuser/Saved Games/CD Projekt Red/Cyberpunk 2077/` (all libraries from `libraryfolders.vdf`)
  - Heroic/Lutris GOG prefixes: best-effort known paths.
  - Always with a manual folder picker fallback.
- **PLT-2 ŌĆö Game-dir detection**: Steam library scan (both OSes), GOG registry keys (Windows), manual fallback. Validated by presence of `archive/pc/content`.
- **PLT-3**: Local filesystem paths use `pathlib` throughout; game depot paths keep Windows backslashes per game convention (unchanged invariant from CLAUDE.md).
- **PLT-4**: `.rar` hair-mod support uses a bundled/pip extraction path where possible; if an external `unrar` is required it is resolved to an absolute, verified location or the feature degrades with a clear message (no silent PATH lookup).

## 4. Functional Requirements

### 4.1 Patch currency (PC) ŌĆö P0

- **PC-1**: `detect_patch()` hard-fails on unknown build numbers with `UnsupportedPatchError` naming the detected build and the supported list. The silent 2.13 default (`save_parser.py:97-110`) is removed.
- **PC-2**: `save_versions.json` gains entries for the 2.2 ŌåÆ 2.31 build numbers (determined empirically from real saves).
- **PC-3**: CC-struct parsing becomes versioned: a decoder registry keyed by the save's `v3` value. `v3=195` (2.13) is kept; the 2.3x struct version is determined empirically from a current save and implemented. Unknown `v3` ŌåÆ hard fail with diagnostics dump option.
- **PC-4**: New `mappings/2.3x.json` and `donors/2.3x.json` covering the 2.2 CC additions (100+ options). A helper script (`tools/gen_mapping.py`) semi-automates table generation by scanning uncooked game data, so future patches are cheaper.
- **PC-5**: WolvenKit.CLI minimum version raised to 8.19.0; `check_version` enforces the minimum (hard fail below it, warn above tested version).
- **PC-6 ŌĆö Bake verification**: after morphtarget import, the pipeline verifies the result (shape-key/diff count via WolvenKit serialize) and hard-fails with `BakeVerificationError` if morphs were dropped (guards WolvenKit issue #849). Blender export recomputes Basis normals and strips custom split normals first.

### 4.2 ArchiveXL spike (AX) ŌĆö P1, decision gate

- **AX-1**: Time-boxed spike (target Ōēż 1 week): author an ArchiveXL `.xl` resource patch + self-authored `.app` that adds V's appearance to an entity AMM can spawn, on patch 2.3x, without binary-editing cooked files.
- **AX-2**: Success criteria: NPV spawns via AMM, full animation rig works (no T-pose), head morphs and clothing render correctly, mod survives a game restart.
- **AX-3 ŌĆö Branch A (spike succeeds)**: pipeline authors `.xl` + `.app`/`.ent` via WolvenKit JSON serialization; `npv-inject` (.NET) and donor-entity patching are removed; `.NET` is dropped from runtime dependencies and the installer.
- **AX-4 ŌĆö Branch B (spike fails)**: current design stays; `npv-inject` is published as self-contained .NET builds for win-x64 and linux-x64 and bundled/downloaded like other tools.
- **AX-5**: The spike outcome and rationale are recorded in `docs/adr/` before dependent work proceeds.

### 4.3 GUI as primary driver (GUI)

- **GUI-1 ŌĆö Feature parity**: everything the CLI can do is doable in the GUI: save selection, NPV naming, garment overrides, BYO head, CET-dump import, hair-mod browser/installer, output/install location.
- **GUI-2 ŌĆö First-run wizard**: locate game dir (PLT-2 auto-detect + manual), dependency check and guided auto-install (WolvenKit, Blender, .NET only in branch B), writes config.
- **GUI-3 ŌĆö Save browser**: lists discovered saves (PLT-1) with metadata (save name, timestamp, screenshot thumbnail from the save folder); manual file picker fallback.
- **GUI-4 ŌĆö Build view**: stage-by-stage progress (from CORE-2/LOG-3 events), live log pane, Cancel button (CORE-4), and on failure a "Retry from failed stage" action (CORE-3).
- **GUI-5 ŌĆö Mod manager**: list NPV mods built/installed by this tool, one-click install/uninstall into the game dir (archive + AMM lua), open-mod-folder.
- **GUI-6 ŌĆö Multi-appearance**: add an additional appearance (from another save or garment set) to an existing NPV mod instead of building a new mod (mirrors the community "add appearance to an NPV" workflow).
- **GUI-7 ŌĆö Settings**: paths (game dir, tools, output), log verbosity, patch override (advanced), update checks toggle.
- **GUI-8 ŌĆö Errors**: all failures surface per ERR-3; no raw tracebacks in the UI.

### 4.4 CLI (secondary surface)

- **CLI-1**: `npv-build` remains, rewired onto `PipelineService`; behavior-compatible flags plus `--resume`, `--log-file`, `--list-saves`.
- **CLI-2**: CLI gains the same preflight dependency checks as the GUI (shared `core` code), fixing the weaker-CLI-preflight gap.

### 4.5 Security (SEC) ŌĆö P2 but release-blocking

- **SEC-1**: One safe-extraction helper validates member paths for zip/tar/7z (and the rar path) before extraction; used by `installer.py` and `hair_mod_helper.py`. Regression tests include crafted zip-slip archives.
- **SEC-2**: Every download is checksum-verified: Blender against blender.org's published SHA-256 lists; WolvenKit and npv-inject against release checksums; the dotnet-install scripts pinned to a known version + hash before execution (branch B only). Verification failure ŌåÆ `SecurityError`, never a silent fallback.
- **SEC-3**: External binaries are invoked by absolute, validated path only.

## 5. Non-Functional Requirements

- **NFR-1 ŌĆö Compatibility matrix**: Windows 10/11 x64 and mainstream Linux x64 (glibc Ōēź AppImage baseline); game patches 2.13 and 2.2ŌĆō2.31; WolvenKit.CLI Ōēź 8.19; Blender 4.x; Python 3.11+ (bundled).
- **NFR-2 ŌĆö Code quality gates** (CI-enforced): `ruff` clean, type check clean on `core/`, tests green on both OS runners.
- **NFR-3 ŌĆö Repo hygiene**: build-artifact dirs (`my_v_mod/`, `latest_v_mod/`, `test_v_mod/`, `trial_out/`, `release_test/`, `external_v_01_mod/`, `app_verify/`) and `venv/` removed from git and ignored; empty stubs removed; legacy specs moved per D6.
- **NFR-4 ŌĆö No CDPR bytes in repo or release artifacts** (unchanged invariant).
- **NFR-5 ŌĆö Determinism**: mod ID remains a stable hash of `(npv_name, cc_settings)` so reinstalls overwrite in place.

## 6. Testing Strategy

- **TST-1 ŌĆö Unit coverage for the untested core**: `save_format.py`, `config.py`, `part_resolver.py`, `wolvenkit.py`/`assemble`, `blender_module.py` (adapter-mocked), plus the new `core/` layer. Orchestrator stub test replaced with real stage-sequencing tests using fake adapters.
- **TST-2 ŌĆö Fixtures**: real save files for 2.13 and 2.3x checked into `tests/fixtures/` (saves are user-generated data, no CDPR assets; personal name fields scrubbed). Golden-file tests for CC extraction per patch.
- **TST-3 ŌĆö Integration (game-install-gated)**: an E2E marker (`pytest -m e2e`) runs the full pipeline against a real install; excluded from CI, run manually on both OSes before release.
- **TST-4 ŌĆö GUI**: `gui_backend`/view-model logic unit-tested; GUI smoke launch in CI (xvfb on Linux, native on Windows runner).
- **TST-5 ŌĆö Security regression tests** per SEC-1/SEC-2.
- **TST-6 ŌĆö CI matrix**: GitHub Actions on `ubuntu-latest` + `windows-latest`: ruff, type check, unit tests, GUI smoke, packaging dry-run.
- **TST-7 ŌĆö Release QA checklist** (manual, both OSes): fresh-machine install ŌåÆ wizard ŌåÆ build from a real 2.3x save ŌåÆ spawn in game via AMM ŌåÆ verify appearance/animation; documented as a checklist in `docs/release-qa.md`.

## 7. Packaging and Release

- **PKG-1**: PyInstaller one-dir builds of the GUI (CLI included in the same bundle as a console entry).
- **PKG-2**: Windows artifact: versioned `.zip` (Inno Setup installer is a later, optional follow-up). Linux artifact: AppImage.
- **PKG-3**: External tools are not bundled in the artifact; the first-run wizard downloads them checksum-verified into the app data dir (respecting NFR-4 and licensing).
- **PKG-4**: Release automation: on git tag `v*`, GitHub Actions builds both artifacts, generates `SHA256SUMS`, and attaches everything to a draft GitHub Release with changelog notes.
- **PKG-5**: Version 2.0.0 (semver); `CHANGELOG.md` maintained.
- **PKG-6**: `uv.lock` committed; dev setup documented (`uv sync`).

## 9. Risks and Open Questions

- **R1 ŌĆö Unknown 2.3x CC struct (`v3`)**: no public documentation; must be reverse-engineered from a real save (CyberCAT is the reference implementation to consult). *Mitigation: PC-3 diagnostics dump; timebox; worst case ship 2.13-only with hard-fail messaging while 2.3x lands in 2.0.1.*
- **R2 ŌĆö Mapping-table effort**: 100+ new CC options to map. *Mitigation: PC-4 generator script; community NPV resources as cross-reference.*
- **R3 ŌĆö ArchiveXL spike fails** ŌåÆ branch B keeps .NET; packaging and installer stay heavier. Spec covers both branches by design.
- **R4 ŌĆö Windows is untested territory** for this codebase (developed on Linux). *Mitigation: CI Windows runner from M0; early manual smoke on Windows.*
- **R5 ŌĆö Blender/WolvenKit round-trip regressions** (WK#849). *Mitigation: PC-6 automated verification.*
- **Q1**: Exact `v3` value(s) for 2.2/2.21/2.3/2.31 ŌĆö resolve in M2 with real saves.
- **Q2**: Whether one mapping table covers all of 2.2ŌĆō2.31 or minor patches need separate tables ŌĆö resolve in M2.
- **Q3**: AppImage vs Flatpak preference among target users ŌĆö AppImage chosen for zero-infra; revisit post-release if users ask.
